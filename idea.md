\### Название концепции: «Адаптивная мультистратегическая маршрутизация»



\#### 1. Проблема

Текущие решения для обхода DPI (например, Zapret) работают по статическим правилам. Пользователь выбирает одну стратегию (профиль), которая применяется ко всему трафику или к заранее заданному списку доменов.

Это приводит к двум основным проблемам:

\*   Универсальность против эффективности: Универсальная стратегия, работающая на большинстве провайдеров, может не сработать на конкретном сайте или быть слишком медленной. Стратегия, идеально разблокирующая YouTube, может сломать авторизацию в банке или работу Discord.

\*   Ручная настройка: Если сайт перестает открываться, пользователю приходится вручную перебирать стратегии, искать рабочую и обновлять списки. Это требует времени и компетентности.



\#### 2. Предлагаемое решение

Создание системы, которая автоматически подбирает и применяет оптимальную стратегию обхода для каждого конкретного домена в реальном времени, не прерывая работу других сервисов.



\#### 3. Архитектура и алгоритм работы



Система состоит из трех основных модулей: Монитор, Анализатор и Исполнитель.



Шаг 1: Обнаружение проблемы (Мониторинг)

Система постоянно анализирует статус сетевых соединений.

\*   Входные данные: Логи Zapret (параметры hostlist-auto-fail-threshold), события таймаута соединения в браузере/ОС, или ручной запрос пользователя через UI («Сайт не работает»).

\*   Триггер: Если соединение с доменом example.com прерывается или таймаутится N раз за короткий промежуток времени, система помечает этот домен как «проблемный».



Шаг 2: Подбор решения (Анализ)

Как только домен помечен как проблемный, запускается процедура поиска решения.

\*   База знаний: Система хранит базу данных о том, какие домены какими стратегиями уже успешно обходятся (например, youtube.com -> Strategy\_A, discord.com -> Strategy\_B).

\*   Логика подбора:

&#x20;   1.  Проверка текущих правил: Есть ли у нас уже работающее правило для этого домена? Если да, возможно, проблема не в DPI (сервер упал, проблемы с DNS).

&#x20;   2.  Тестирование активных стратегий: Система по очереди применяет к домену уже запущенные (активные) стратегии, которые используются для других сайтов, и проверяет доступность.

&#x20;   3.  Расширенный поиск (если п.2 не помог): Запускается перебор всех доступных стратегий из библиотеки (в фоновом режиме или с низким приоритетом), пока не будет найдена рабочая.



Шаг 3: Применение решения (Исполнение)

Найденная рабочая стратегия применяется к домену без перезапуска основной службы.

\*   Механизм: Используется функционал Zapret по работе с множественными стратегиями (разделитель --new в конфигурации nfqws/winws).

\*   Динамическое обновление: Система генерирует и «подгружает» новое правило в оперативную память работающего процесса (или через перезагрузку конфига без сброса сессий), добавляя домен в hostlist соответствующей стратегии.

\*   Результат: Для пользователя example.com начинает открываться, в то время как остальной трафик продолжает обрабатываться по прежним правилам.



\#### 4. Ключевые требования к реализации



Чтобы система работала так, как вы задумали (без перебоев), должны быть выполнены условия:



1\.  Независимость стратегий: Движок (Zapret) должен уметь одновременно обрабатывать несколько разных правил для разных списков доменов. Это уже реализовано через --new в командной строке, но требует автоматизации управления этими списками.

2\.  Горячее применение: Добавление нового домена в список (хост-лист) работающей стратегии должно происходить без полного перезапуска службы и разрыва текущих соединений (например, отправка сигнала HUP для перечитывания конфигурации или использование API, если он появится).

3\.  Фоновый режим тестирования: Процесс подбора стратегии для нового домена не должен блокировать интернет или нагружать процессор на 100%, мешая просмотру видео или играм.



\#### 5. Пользовательский сценарий (Use Case)

Пользователь пытается зайти на newsite.com, который его провайдер начал блокировать по новой сигнатуре.

2\.  Браузер показывает ошибку, но «Адаптивный модуль» это замечает (по повторным ретрансмиссиям или таймаутам).

3\.  В фоновом режиме система запускает тестирование newsite.com через другие доступные стратегии.

4\.  Через 10–15 секунд находится рабочая стратегия (например, с параметром --dpi-desync=split2).

5\.  Система автоматически создает правило: «Для домена newsite.com использовать Strategy\_X».

6\.  Пользователь обновляет страницу — сайт открывается. Вмешательство пользователя не потребовалось.



\#### 6. Итог

Ваша идея заключается в переходе от статической модели (один конфиг на всех) к динамической, самонастраивающейся системе. Это аналог «умного антиспама» или адаптивного фаервола, но для обхода цензуры, где система сама определяет наилучший метод доставки контента для каждого конкретного ресурса.
