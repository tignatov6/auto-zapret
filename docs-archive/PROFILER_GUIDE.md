# Профилирование производительности

Все функции и методы в проекте Auto-Zapret обернуты в профайлер для измерения производительности.

## Структура

- `autozapret/utils/profiler.py` - модуль профайлера
- `autozapret/helpers.py` - общие утилиты (normalize_domain, canonicalize_params)
- Все остальные модули декорированы через `@profiler`

## Как это работает

Каждая функция декорирована `@profiler`, который автоматически измеряет:
- **Общее время выполнения** (суммарное время всех вызовов)
- **Количество вызовов**
- **Процент от общего времени** для каждого модуля

**Поддерживаются:**
- ✅ Синхронные функции (`def`)
- ✅ Асинхронные функции (`async def`)
- ✅ Методы классов
- ✅ Менеджеры контекста (`with profiler("name")`)

## Использование

### Программный вызов отчёта

```python
from autozapret.utils.profiler import report

# Вывести отчёт (сортировка по времени)
report(sort_by="time")

# Или сортировка по количеству вызовов
report(sort_by="calls")
```

### Включение/отключение профилирования

По умолчанию профилирование **включено**. Для отключения:

```python
from autozapret.utils.profiler import enable_profiling

# Отключить профилирование (для продакшена)
enable_profiling(False)

# Включить обратно
enable_profiling(True)
```

### Проверка статуса

```python
from autozapret.utils.profiler import is_profiling_enabled

if is_profiling_enabled():
    print("Профилирование включено")
else:
    print("Профилирование отключено")
```

### Очистка статистики

```python
from autozapret.utils.profiler import clear_stats

# Очистить всю собранную статистику
clear_stats()
```

## Интерпретация результатов

### На что обращать внимание

1. **Модули с наибольшим общим временем** (`total X.XXX s`)
   - Показывает какие части программы потребляют больше всего времени
   - Помогает найти "узкие места" для оптимизации

2. **Функции с большим процентом** (`XX.X%`)
   - Функции, которые доминируют в своём модуле
   - Кандидаты на оптимизацию

3. **Функции с большим количеством вызовов** (`XXX×`)
   - Частые вызовы могут указывать на неоптимальные алгоритмы
   - Даже быстрая функция может замедлять систему при частых вызовах

4. **Соотношение время/вызовы**
   - `время / количество` = среднее время одного вызова
   - Помогает найти "тяжёлые" операции

### Пример анализа

```
[executor]  total 5.234 s
  start_winws_with_strategy:   3.123s  |  59.7% |      42×
```

**Вывод:**
- `start_winws_with_strategy` потребляет 59.7% времени модуля
- Среднее время одного вызова: 3.123s / 42 = 74ms
- Это запуск процесса winws - нормальное время для создания процесса
- Оптимизация: кэширование процесса вместо перезапуска

## Отладка

### Просмотр статистики в реальном времени

```python
from autozapret.utils.profiler import report, _STATS

# Просмотр "сырой" статистики
print(_STATS)

# Красивый отчёт
report()
```

### Локальный профайлер для воркеров

```python
from autozapret.utils.profiler import get_local_profiler, merge_stats

# Создание локального профайлера для воркера
local_stats = {}
local_lock = threading.Lock()
worker_profiler = get_local_profiler("worker", local_stats, local_lock)

# ... использование worker_profiler ...

# Слияние с глобальной статистикой
from autozapret.utils.profiler import merge_stats
merge_stats(local_stats)
```

## Производительность

Профайлер добавляет минимальные накладные расходы:
- ~1-2 микросекунды на вызов функции
- Использует `time.perf_counter()` для высокой точности
- Потокобезопасен (thread-safe)
- Может быть отключен в продакшене через `enable_profiling(False)`

## Структура отчёта

```
[module_name]  total X.XXX s
  function_name:                X.XXXs  |  XX.X% |    N×
```

Где:
- `module_name` - имя модуля (файла)
- `total` - суммарное время всех функций модуля
- `function_name` - имя функции/метода
- `X.XXXs` - общее время выполнения функции
- `XX.X%` - процент от общего времени модуля
- `N×` - количество вызовов

## Декорирование функций

Все функции в проекте автоматически декорированы при инициализации:

```python
from .utils.profiler import get_profiler

profiler = get_profiler("module_name")

@profiler
def my_function():
    ...

@profiler
async def my_async_function():
    ...

class MyClass:
    @profiler
    def my_method(self):
        ...
    
    @profiler
    async def my_async_method(self):
        ...
```

## Советы по оптимизации

1. **Ищите функции с >50% времени модуля** - главные кандидаты на оптимизацию
2. **Смотрите на функции с 1000+ вызовами** - даже микрооптимизация даст эффект
3. **Проверяйте I/O операции** (диск, сеть, БД) - часто имеют наибольший потенциал
4. **Сравнивайте отчёты до/после** изменений для оценки эффективности оптимизаций

## Примеры использования

### Быстрая проверка производительности

```python
import asyncio
from autozapret import main
from autozapret.utils.profiler import report, clear_stats

async def benchmark():
    clear_stats()  # Очистка перед тестом
    
    # ... выполнение тестируемого кода ...
    
    report()  # Вывод статистики

asyncio.run(benchmark())
```

### Сравнение двух подходов

```python
from autozapret.utils.profiler import clear_stats, report

# Тест 1
clear_stats()
# ... код варианта 1 ...
print("=== Вариант 1 ===")
report()

# Тест 2
clear_stats()
# ... код варианта 2 ...
print("=== Вариант 2 ===")
report()
```

## Вопросы и ответы

**В: Насколько профайлер замедляет работу?**
О: ~1-2 мкс на вызов функции. Для I/O операций это незаметно, для CPU-bound кода может быть заметно при миллионах вызовов.

**В: Можно ли профилировать только часть кода?**
О: Да, используйте `enable_profiling(False)` для отключения, затем включайте перед нужным участком.

**В: Как собрать статистику в многопоточном приложении?**
О: Профайлер потокобезопасен. Для воркеров используйте `get_local_profiler()` и `merge_stats()`.

**В: Почему некоторые функции показывают 0.000s?**
О: Они выполняются быстрее разрешения таймера (<1 мкс) или не вызывались.
